# vino_platform 落地执行计划

- Product: `vino_platform`
- Date: 2026-04-29
- Status: Draft

## 执行原则

- `vino_iPhone` 本轮不动，只按现有云端接口接入。
- 先完成模型交易到终端下载的最小闭环。
- 支付、发票、结算先人工兜底，再自动化。
- 模型资产、授权、下载和审计必须从第一天按生产要求设计。
- `vino_cloud` 代码只作为 MVP 参考，不直接承担正式交易平台。

## 阶段 0：需求冻结与技术底座，1 周

目标：

- 冻结 P0 范围、数据模型、API 兼容层。
- 确定后端、前端、数据库、对象存储、队列、部署方式。
- 把 `vino_cloud` 的下载与授权逻辑拆成可迁移清单。

交付：

- 数据库 schema 初稿。
- API OpenAPI 草案。
- Web 管理后台信息架构。
- 文件存储和加密方案。

验收：

- iPhone 终端所需 5 个接口字段冻结。
- P0 后台页面清单冻结。
- 模型文件生命周期和授权生命周期冻结。

## 阶段 1：Identity、Catalog、Delivery，2 周

目标：

- 完成账号、组织、角色、模型目录、模型构建、下载服务。
- 支持管理员上传 CoreML 模型并生成构建记录。

任务：

- 后端：Organization、User、Session、Model、ModelBuild。
- 文件：对象存储上传、sha256、大小、格式识别。
- 下载：下载票据、短时 URL、AES-GCM 加密包。
- 前端：管理员登录、模型列表、模型上传、构建详情。

验收：

- 管理员能上传 `.mlpackage` 并看到哈希和构建状态。
- 终端能登录并拿到空模型列表。
- 生成的下载包格式与 `vino_iPhone` 当前 `ModelFileStore` 兼容。

## 阶段 2：Entitlement、Lease、Device，2 周

目标：

- 完成授权分配、设备绑定、离线租约和续租。

任务：

- 后端：Entitlement、Device、OfflineLease、DownloadTicket。
- 权限：组织级、用户级、设备级授权。
- 终端兼容：`/api/cloud/v1/models`、`download-ticket`、`lease/renew`。
- 后台：授权列表、创建授权、撤销授权、设备列表。

验收：

- 未授权用户看不到模型。
- 授权用户能下载并安装模型。
- 票据过期后不可下载。
- 设备绑定不匹配时无法续租或下载。

## 阶段 3：订单与最小交易闭环，3 周

目标：

- 支持采购方下单，运营确认收款后自动生成授权。

任务：

- 后端：ModelSKU、Order、OrderItem、Payment。
- 后台：SKU 配置、订单管理、手工确认收款、退款标记。
- 采购方 Web：模型列表、模型详情、创建订单。
- 授权：订单支付成功后生成 Entitlement。

验收：

- 一个模型 SKU 可被采购方下单。
- 运营确认收款后，系统自动生成授权。
- iPhone 登录后可见该订单对应模型并可下载。

## 阶段 4：开发者、审核与运营后台，3 周

目标：

- 支持开发者入驻、模型提交审核、平台审核、上下架和审计。

任务：

- 后端：Developer、Qualification、ReviewCase、AuditLog。
- 开发者 Web：入驻资料、模型草稿、上传构建、提交审核。
- 审核后台：模型审核、资质审核、驳回原因、上下架。
- 运营后台：用户、开发者、模型、订单、授权、设备统一查询。

验收：

- 开发者可提交模型。
- 审核通过前模型不可售卖。
- 审核、上下架、授权、撤销均有审计日志。

## 阶段 5：安全、测试与试运行，2 周

目标：

- 打磨安全、稳定性、性能和试运行问题。

任务：

- 安全：权限矩阵、下载限流、异常下载告警、敏感日志脱敏。
- 测试：端到端、负向授权、安全、并发下载、支付回调模拟。
- 运维：备份、监控、审计导出、错误追踪。
- 试运行：导入 3 到 5 个真实 CoreML 模型，邀请试点组织测试。

验收：

- 端到端主链路通过。
- 关键负向场景通过。
- 试点企业可以完成模型购买到终端下载。
- 平台运营可以追踪每次授权和下载。

## 风险清单

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| App Store 审核把外部购买识别为 App 内数字内容解锁 | 上架风险 | 定位 B2B 企业服务，终端只做授权同步，不做商城导购 |
| 模型文件被复制 | 开发者信任下降 | 加密包、设备绑定、短租约、审计、合同追责 |
| 支付和结算复杂 | 延误 P0 | P0 线下收款人工确认，P1 再自动支付和结算 |
| 模型质量差 | 平台口碑受损 | 审核、试用、评价、售后、开发者等级 |
| 授权规则过复杂 | 工程延期 | P0 只做组织、用户、设备三类授权 |
| 大模型文件下载失败 | 终端体验差 | 断点续传列入 P1，P0 先支持重试和票据重发 |
