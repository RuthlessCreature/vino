## 结论

`vino_platform` 应收敛为一个面向工业视觉模型的 B2B 模型交易与授权平台。它的核心价值不是“展示很多 AI 模型”，而是把模型从开发者交付到企业现场终端的全过程做成可交易、可审核、可授权、可下载、可续期、可结算、可追溯的闭环。

首发产品形态：

- Web 平台：采购、开发者、平台运营后台。
- 云端服务：账号、组织、模型、订单、支付、授权、下载票据、离线租约、审计。
- 终端接入：`vino_iPhone` 只作为工业现场采集与推理终端。
- 参考实现：`vino_cloud` 的模型发现、授权分配、下载票据、AES-GCM 加密包、离线租约、ingest 端点。

## MVP 要赢什么

第一版不要追求完整大而全交易市场，而要先赢下一个闭环：

1. 平台管理员创建企业、用户、开发者。
2. 开发者或平台上传一个 CoreML 模型包。
3. 平台审核模型并配置价格和授权策略。
4. 采购方完成订单，或运营方通过线下合同录入订单。
5. 系统生成 Entitlement，并分配给组织、用户或设备。
6. `vino_iPhone` 登录后只看到已授权模型。
7. 终端申请下载票据，下载加密模型包，校验哈希后安装。
8. 离线期内可运行，到期后需要联网续租。
9. 平台能查到订单、授权、设备、下载、运行结果和审计日志。

## 成功指标

| 指标 | MVP 目标 |
| --- | --- |
| 授权闭环 | 一个企业账号可以从订单生成授权，并让 iPhone 下载模型成功 |
| 下载可靠性 | 500 MB 以内模型下载、校验、安装成功率不低于 95% |
| 授权安全 | 未授权账号无法看到或下载模型；过期授权无法续发下载票据 |
| 运营效率 | 平台运营可在 5 分钟内完成模型上架到授权分配 |
| 审计完整性 | 模型上传、审核、授权、下载、续租、删除均有审计记录 |

## 首发范围

P0 必须交付：

- 组织、用户、角色权限。
- 开发者入驻的最小审核流程。
- 模型上传、模型构建记录、模型审核、模型目录。
- 模型 SKU、价格、授权策略。
- 订单录入和支付状态管理，先支持线下收款/人工确认。
- Entitlement 分配到组织、用户、设备。
- iPhone 兼容接口：登录、模型列表、下载票据、模型下载、租约续期。
- 加密模型分发：沿用 `vino_cloud` 的 `vino-aesgcm-v1` 思路。
- 管理后台：用户、模型、订单、授权、设备、审计。

P1 再交付：

- 在线支付、退款、结算、提现、发票。
- 模型在线试用。
- 开发者收入看板。
- 工单、售后、评价。
- 本地 Web ingest 节点与云端归档的生产化。

## 关键假设

- 采购方以企业客户、集成商、工厂项目团队为主。
- 模型首发形态以 CoreML 工业视觉模型为主。
- `vino_iPhone` 的现有云端登录与下载协议可以作为终端兼容基线。
- 平台早期可以用人工审核、线下合同、线下付款兜底，先验证交易和授权闭环。

## 最大风险

- App 内外部购买与数字内容解锁的审核边界需要按 B2B 企业服务叙事处理。
- 模型 DRM 只能提高盗版成本，无法绝对防止被提取。
- 交易、结算、发票和税务流程上线后会显著提高运营复杂度。
- 模型质量和版权审核如果没有标准，会变成平台信用风险。

## 下一步

冻结 P0 范围后，先搭 `vino_platform` 的后端数据模型、API 网关和管理后台骨架，再迁入 `vino_cloud` 的模型下载与租约链路。
