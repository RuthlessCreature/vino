# vino 商业逻辑汇报

> 老板版  
> 日期：2026-04-06  
> 目标：讲清楚 `vino` 卖什么、为什么能卖、怎么赚钱、为什么现在该这样做

## 封面

一句话定义：

- `vino` 不是一个桌面软件项目，而是一个面向工业现场的“iPhone 执行端 + 本地接收节点 + 云端业务平台”。
- 真正要卖的不是 App 本身，而是“平台能力 + 模型能力 + 交付服务”。

## 1. 我们到底在卖什么

- 面向工业场景的移动视觉采集与推理系统
- iPhone 负责现场采集、推理、任务执行
- 本地 Web 节点负责接收图片、视频、结果并做缓存转发
- 云端 Web 负责账号、授权、模型售卖、项目管理和结果查看
- 商业本质是：`工业视觉平台 + 模型授权 + 服务交付`

## 2. 客户为什么会买

- 现场客户需要低成本、低门槛的视觉采集和推理能力
- 传统工业相机 + 工控机 + 本地软件的部署成本高、迭代慢
- 纯云方案不适合现场网络不稳定、对时延敏感的环境
- 纯本地方案又难以统一管理、授权和升级
- `vino` 把“现场可用”和“平台化管理”结合到了一起

## 3. 产品形态为什么要这样收敛

- `Desktop` 不是必须的商业核心
- 真正关键的是：
  - iPhone 能执行
  - 本地节点能稳定收数据
  - 云端能做业务管理和授权
- 这样产品更轻、更容易卖、更适合规模化
- `Desktop` 最多保留为内部调试或大客户运维工具

## 4. 业务闭环怎么跑通

- 第一步：客户在云端后台购买平台和模型授权
- 第二步：账号登录 iPhone，下载已授权模型
- 第三步：iPhone 在现场执行采图、录制、推理
- 第四步：数据先进入本地 Web 节点，保证现场稳定接收
- 第五步：本地节点再把数据补传到云端
- 第六步：客户在云端查看项目、结果、设备和报表

## 5. 我们怎么赚钱

- 平台费
  - 按组织、站点、设备席位或年费收费
- 模型费
  - 通用模型
  - 行业模型
  - 定制模型
- 服务费
  - 部署实施
  - 调参优化
  - 定制训练
  - 售后支持 / SLA

## 6. 授权应该怎么卖

- 不建议只卖“账号权限”
- 建议按四个维度授权：
  - 组织
  - 设备
  - 模型
  - 时间
- 可售卖的授权形态：
  - 年费订阅
  - 项目期授权
  - 永久授权 + 年维护
  - 站点授权 / 设备包

## 7. 为什么“本地接收 + 云端业务”更有商业价值

- 更容易落地到真实工业现场
- 对客户更有说服力：断网也能先收，联网后再同步
- 平台能力放在云端，便于持续收费和统一运营
- 本地节点增强交付稳定性，降低现场出问题的概率
- 这条路线既能做标准化产品，也能接高客单的项目型收入

## 8. 关键风险与边界

- App Store 对外部购买后在 App 内使用仍有审核边界
- 模型离线使用必须有授权与续期逻辑，不能无限离线
- 本地节点与云端之间需要可靠的补传和幂等
- 模型可以做强保护，但做不到绝对防拷贝
- 所以产品定位必须坚持 B2B 工具，而不是消费型模型商城

## 9. 这件事的老板视角价值

- 它不是单次项目，而是有机会形成平台收入
- 它不是只卖软件，而是“平台 + 模型 + 服务”的复合收入
- 它不是只能做一个行业，后续可以扩多个模型场景
- 它的关键护城河不只是代码，而是：
  - 授权体系
  - 交付能力
  - 行业模型
  - 数据闭环

## 10. 现在最需要拍板的事

- 是否正式放弃把 `Desktop` 作为对外主产品
- 是否确定首发架构为：
  - `iPhone 执行端`
  - `本地 Web 接收节点`
  - `云端 Web 业务平台`
- 是否确定商业模式为：
  - 平台订阅
  - 模型授权
  - 服务收费
- 是否优先做 B2B，不做消费级模型商店

## 11. 下一阶段建议

- 先做老板能卖的故事，而不是先做最复杂的技术
- 下一阶段文档和产品重点：
  - 本地 Web 接收节点 PRD
  - 云端授权与模型商城 PRD
  - iPhone 登录、下载、离线授权方案
  - 上传 API 与补传机制
- 目标不是“功能很多”，而是先形成一条能卖、能交付、能续费的主线
