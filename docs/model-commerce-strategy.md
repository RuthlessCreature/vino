# vino 模型售卖、账号授权与离线使用方案

> 版本：`v0.1`  
> 日期：`2026-04-06`  
> 适用范围：`vino_iPhone`、未来 Web 模型商城、账号体系、授权服务

## 1. 结论先行

如果 `vino` 要做“Web 售卖模型，iPhone 登录账号后下载并使用”的模式，**最稳妥的路线不是把它当成消费级模型商店，而是把它定位成 B2B 工业视觉工具 + 账号化模型授权系统**。

推荐结论：

- `vino_iPhone` 继续作为工业采集/推理终端，而不是 App 内商城。
- `vino_Desktop` 不应再作为首发商业化核心，可以降为内部调试、演示或运维工具。
- 购买动作优先发生在 Web 后台或销售流程中。
- iPhone 端只负责：
  - 登录账号
  - 同步可用模型 Entitlement
  - 下载已获授权模型
  - 在离线期内按授权规则使用
- 服务端负责：
  - 本地 Web：接收图片、视频、推理结果，做缓冲、转发、局域网落盘
  - 云端 Web：保存业务数据、授权数据、项目数据和最终归档
  - 云端服务：提供账号、订单、授权、模型目录和业务后台
- 商业模式优先选择：
  - 平台订阅 + 模型授权
  - 企业账户 + 设备席位
  - 永久授权 + 年度更新维护
  - 限时授权 / 项目授权 / 现场授权

---

## 2. 为什么这条路线更适合 `vino`

`vino` 当前仓库已经更像一个工业现场系统，而不是面向普通消费者的内容 App：

- iPhone 端负责采集、推理、参数控制、模型安装
- Desktop 端负责发现、批控、模型分发、结果管理
- 模型本身是“生产工具能力”，不是娱乐内容
- 实际客户更像工厂、集成商、项目团队、部署服务商

这意味着：

- 商业上更适合做 `B2B`，而不是做大量零售散客
- 鉴权上更适合做 `组织 / 设备 / 项目` 维度授权
- App Store 合规上更适合走 `企业服务` 或 `付费 Web 工具的伴随 App` 路线

## 3. App Store 约束与可行边界

以下判断基于我在 **2026-04-06** 查看的 Apple 官方资料：

- Apple `App Review Guidelines` 当前仍写明：如果是在 App 内解锁功能、能力或数字内容，通常需要使用 `In-App Purchase`。
- `Reader App` 例外主要适用于杂志、书籍、音频、视频等内容，不适合 `vino` 这种工业模型分发场景。
- **2025-05-01** Apple 更新了美国区 storefront 的外链规则，允许美国区 App 出现外部购买方式相关按钮或链接，但这并不等于“任意全球消费型模型商店都天然安全”。
- Apple 当前的 `4.7.2` 还明确限制：App 不得在未经许可的情况下把原生平台 API 或技术能力暴露给“非二进制内置的软件”。**我的推断**是：`vino` 下载的对象应被明确描述为受控的 CoreML 模型资产，而不是通用插件、脚本或小程序。
- Apple 还保留了：
  - `3.1.3(c) Enterprise Services`
  - `3.1.3(f) Free Stand-alone Apps`
  这两类对 `vino` 更有现实意义。

### 3.1 对 `vino` 的实际含义

最安全的产品/审核叙事是：

- `vino_iPhone` 是企业或专业用户的现场工作终端
- 用户登录的是企业/组织账户
- 模型授权来自组织购买或付费 Web 工具后台分配
- App 内不做显式消费型商城，不做强销售导流

### 3.2 风险最高的路线

风险最高的是把它做成：

- 消费者下载免费 App
- App 外网站随意买模型
- App 内直接像“数字内容商城”一样下载启用

这种模式更容易被 Apple 认定为：

- 数字内容 / 功能在 App 内被外部付款解锁
- 不满足 `Reader App`
- 不足以构成明确的企业服务或伴随工具

### 3.3 推荐审核定位

推荐把 `vino` 对外定义为：

- 工业视觉采集与推理工具
- 面向企业项目和专业用户
- iPhone App 为现场执行终端
- Web 为模型配置、授权、项目管理和订单分配后台
- 下载模型仅作为受控推理资产使用，不作为通用扩展程序生态

这比“模型商店 App”安全得多。

## 4. 推荐商业模式

## 4.1 推荐主模式：平台订阅 + 模型授权

这是我最推荐的起步方案。

### 平台层

组织先购买平台使用权，例如：

- 按组织订阅
- 按现场/产线订阅
- 按设备席位订阅

平台层包含：

- iPhone App 使用权
- Desktop 控制台使用权
- 基础账号体系
- 基础模型管理与同步能力
- 日志、项目、设备管理

### 模型层

模型作为附加授权出售：

- 单模型永久授权
- 单模型年费授权
- 项目期限授权
- 多模型组合包
- 企业站点授权

### 这种模式的好处

- 收入结构更稳定
- 不把所有收入都压在一次性买断
- 平台和模型价值可以分别定价
- 更适合 B2B 销售与后续增购

## 4.2 永久授权怎么定义

“永久”建议不要定义成“永久免费更新全部版本”，而应该定义成：

- 永久使用已购买版本或已购买大版本
- 默认含 `12个月` 更新维护期
- 维护期内可下载新版模型构建
- 维护期结束后：
  - 已下载版本仍可继续用
  - 但不能获取新版本模型
  - 也不能继续获得新的离线延长期策略（可选）

这样既满足客户对“买断”的心理预期，又不会变成无限期维护义务。

## 4.3 限时授权怎么定义

限时授权建议分三类：

- `月度/年度订阅`
- `项目期授权`：如 `3个月 / 6个月 / 12个月`
- `临时现场授权`：如某次出差、验线、驻场期间有效

限时授权适合：

- 客户先试运行
- 季节性项目
- 外包/集成商按项目结算

## 4.4 席位与设备授权建议

建议不要只做“账号级授权”，而是至少同时支持以下维度：

- `组织`
- `用户`
- `设备`
- `模型`
- `时间`

推荐许可模型：

- `Named Device`：每个授权绑定 1 台 iPhone
- `Floating Seat`：组织买 5 个并发席位，可在多台设备间流转
- `Site License`：某个工厂/产线无限设备
- `Field License`：支持更长离线期的现场授权

## 5. 推荐收入结构

## 5.1 起步版本建议

起步建议不要只卖模型，建议卖三层：

### A. 平台费

- 组织基础使用费
- 覆盖账号、设备、同步、Desktop、日志、远程配置

### B. 模型费

- 通用模型单独收费
- 行业模型单独收费
- 定制模型更高价

### C. 服务费

- 部署调参
- 定制训练
- 现场支持
- SLA/维护

## 5.2 价格结构建议

以下是**推导建议**，不是市场事实：

- 平台订阅：按组织/站点/月或年收取
- 模型授权：
  - 通用模型：低价
  - 行业模型：中价
  - 定制模型：高价
- 永久授权：
  - 一次性付费 + 年维护费
- 限时授权：
  - 年费通常定价为永久授权的较低比例，适合扩大试用

更关键的是价格结构，而不是一开始把绝对价格定死。

## 6. 推荐产品形态

## 6.1 Web 端做什么

Web 端建议拆成两层：

- `本地 Web`
- `云端 Web`

## 6.1.0 本地 Web 做什么

本地 Web 建议承担：

- 接收 iPhone 上传的图片、视频、推理结果
- 作为现场局域网中的低延迟接收入口
- 在本地磁盘或本地对象存储中做短期缓存
- 断网时继续接收并排队
- 网络恢复后向云端补传
- 提供简单的本地任务下发/心跳查看能力

它本质上更像：

**边缘接收网关 / 现场中转站**

## 6.1.1 云端 Web 做什么

云端 Web 建议承担：

- 模型目录展示
- 订单购买
- Entitlement 分配
- 组织账号与设备管理
- 设备激活与解除激活
- 模型版本管理
- 离线授权续期
- 审计日志

云端 Web 本质上不是“普通官网”，而是：

**模型授权控制台 + 商业后台**

## 6.1.2 如果去掉 Desktop，双层 Web 各自承担什么

如果你决定不把 Desktop 作为正式产品，而只是保留 `POST` 控制和上传，那么双层 Web 需要这样分工：

- 本地 Web：
  - 媒体接收接口
  - 推理结果接收接口
  - 上传重试与幂等
  - 局域网缓存
  - 断网续传
- 云端 Web / 云端服务：
  - 账号与授权
  - 模型目录与分发
  - 项目、组织、设备后台
  - 最终归档与检索
  - 报表、审计、业务管理

也就是说，产品结构会从：

- `iPhone + Desktop`

收敛为：

- `iPhone App + 本地 Web 接收节点 + 云端 Web 业务后台`

## 6.2 iPhone 端做什么

iPhone 端只承担：

- 登录
- 查看“我可使用的模型”
- 下载/删除模型
- 本地启用模型
- 看到授权状态和离线到期时间

不建议 iPhone 端一开始就承担：

- 完整购买流程
- 支付结算
- 促销商城逻辑

## 6.3 我更推荐的首发架构

如果你现在确定 Desktop 不是核心，我更推荐首发架构直接定成：

### 端侧

- `vino_iPhone`
  - 登录
  - 下载模型
  - 本地推理
  - 接收远程任务
  - 上传图片/视频/结果

### 本地节点

- `Local Web Ingest`
- `Local Queue / Cache`
- `Local File Store`
- `Cloud Forwarder`

### 云端

- `Auth / License Service`
- `Model Catalog Service`
- `Cloud Control API`
- `Object Storage`
- `Metadata DB`
- `Cloud Web Console`

### 可选内部工具

- `vino_Desktop`
  - 调试控制台
  - 本地联调工具
  - 大批量局域网现场运维工具

这样一来，真正卖给客户的是：

- 云端 Web 平台
- iPhone 端执行 App
- 模型授权
- 可选本地接收节点能力

而不是桌面软件授权。

## 7. 账号、鉴权与授权对象设计

## 7.1 核心对象

建议后端至少有这些实体：

- `Organization`
- `User`
- `Device`
- `Model`
- `ModelBuild`
- `Order`
- `Entitlement`
- `Activation`
- `OfflineLease`
- `RevocationEvent`
- `AuditLog`

## 7.2 它们的关系

- 组织购买模型
- 组织把模型授权分配给用户、设备或站点
- iPhone 登录后，只能看到自己组织可用的模型
- 下载模型前，设备要先完成激活
- 每个模型安装后都要携带可校验的本地离线租约

---

## 8. 推荐授权流程

## 8.1 Web 购买流程

1. 用户在 Web 登录组织账号  
2. 选择模型 SKU  
3. 选择授权类型：永久 / 年费 / 项目期 / 席位数  
4. 订单支付成功  
5. 系统生成 `Entitlement`  
6. 管理员将授权分配到用户或设备池

## 8.2 iPhone 首次使用流程

1. 用户在 iPhone App 登录  
2. App 获取用户、组织、设备状态  
3. App 向服务端请求可见模型列表  
4. 服务端返回：
   - 模型元数据
   - 当前授权状态
   - 是否允许下载
   - 离线期策略
5. 用户点击下载  
6. App 发起设备激活或激活校验  
7. 服务端发放：
   - 短时下载地址
   - 模型构建哈希
   - 授权签名
   - 离线租约
8. App 下载并本地校验后安装

## 8.3 离线使用流程

1. App 启动或推理前检查本地租约  
2. 若租约仍有效，则允许使用  
3. 若租约即将到期，提示联网续期  
4. 若租约过期：
   - 可进入只读状态
   - 或禁止新任务执行
   - 或进入短暂宽限期

---

## 9. 离线授权策略建议

## 9.1 不建议做成永久离线无限期

如果做“买断后永久离线无限使用”，风险很高：

- 设备丢失后难以回收
- 被拷出的模型难以控制扩散
- 企业客户离职/换机时无法回收席位

## 9.2 推荐做“离线租约”

即使客户买的是永久授权，也建议本地运行权限采用：

- 长期许可证 + 短周期离线租约

例如：

- 永久授权：本地离线租约 `30天`
- 年费授权：本地离线租约 `7~30天`
- 现场授权：本地离线租约 `90天`

### 这个设计的好处

- 用户体验上仍然“可离线”
- 商业和风控上保留回收能力
- 授权撤销和到期可以在线后生效

## 9.3 离线到期后的建议策略

建议分三级：

- `提醒期`：剩余 7 天开始提示
- `宽限期`：过期后 24~72 小时有限继续使用
- `失效期`：禁止新推理，仅保留查看或重新登录续期

## 10. iPhone 端推荐安全方案

## 10.1 登录

建议使用：

- 自有账号 + 邮箱验证码
- 或 `Sign in with Apple`
- 或面向企业客户的 SSO

## 10.2 设备身份

建议每台设备首次激活时生成本地密钥对，并把公钥上传服务端。

可结合：

- `Keychain`
- `Secure Enclave`
- `App Attest`
- `DeviceCheck`

### 作用

- 服务端确认是真实 App、真实设备、真实安装实例在请求授权
- 防止纯脚本伪造下载授权请求

## 10.3 本地存储

建议：

- 登录凭证、刷新 Token：放 `Keychain`
- 离线租约：签名后存本地，同时记录到 Keychain 或受保护目录
- 模型文件：放 App 自己的受保护目录
- 每次加载模型前验证：
  - 文件哈希
  - 授权签名
  - 租约有效期
  - 设备绑定是否一致

## 10.4 需要接受的现实

在 iPhone 上做模型 DRM，**可以提高盗版成本，但无法做到绝对防拷贝**。

所以目标应定义为：

- 防普通复制
- 防低成本滥用
- 支持授权撤销
- 支持设备丢失回收

而不是追求“模型文件永远无法被提取”。

## 11. 推荐授权令牌结构

建议服务端给 iPhone 发放一个签名的本地授权对象，至少包含：

- `licenseId`
- `organizationId`
- `userId`
- `deviceId`
- `modelId`
- `modelBuildId`
- `entitlementType`
- `issuedAt`
- `expiresAt`
- `maxOfflineUntil`
- `policyFlags`
- `modelHash`

建议由服务端用私钥签名，App 仅持有公钥验签。

## 12. 永久授权与限时授权的业务规则建议

## 12.1 永久授权

建议定义为：

- 对某一模型或某一主版本永久可用
- 默认含 `12个月` 更新维护
- 可在限定设备数内激活
- 离线租约可续期

## 12.2 限时授权

建议定义为：

- 到期后不可再续发离线租约
- 可在到期前正常离线使用
- 若租约长于合同到期时间，则租约上限不能超过合同结束时间

## 12.3 设备迁移

建议支持：

- 管理后台手动解绑设备
- 冷却期后重新绑定
- 异常场景由管理员强制回收

## 13. 我最推荐的首发方案

如果现在就要做第一版，我建议：

### 商业上

- `vino` 定位成 B2B 工业工具
- 平台按组织收费
- 模型按“年费授权 + 少量永久授权”组合销售
- 定制模型与现场支持另收费

### 产品上

- 本地 Web 做图片/视频/结果接收、缓存、转发
- 云端 Web 做购买、授权分配、项目查看、业务管理
- iPhone 做登录、同步、下载、推理、上传
- 不在 App 内做商城导购
- Desktop 降为可选内部工具，不作为首发售卖主件

### 鉴权上

- 账号登录
- 组织级 Entitlement
- 设备激活
- 7~30 天离线租约
- 服务端签名授权
- 本地哈希校验
- 后续再补 App Attest

### 为什么这样最适合第一版

- 商业结构清晰
- 审核风险更低
- 工程复杂度可控
- 方便以后再扩到：
  - 模型订阅
  - 试用
  - 批量席位
  - 企业私有化
